## user‑stories.md  

### Epic 1 – **AI‑Usage Declaration & Metadata Capture**  
| # | User Story | Acceptance Criteria | Complexity |
|---|------------|----------------------|------------|
| 1 | **As a product manager, I want to embed a standardized AI‑usage manifest into my build pipeline, so that every release automatically records which models, APIs, and data sources were used.** | - A YAML/JSON manifest schema is generated at build time.<br>- Manifest includes model name, version, provider, inference endpoint, and data‑privacy flags.<br>- Manifest is committed to the artifact repository and signed with the company’s CI signing key.<br>- Build fails if the manifest is missing or malformed.<br>- Manifest can be queried via a CLI command (`ai‑usage‑verify show`). | M |
| 2 | **As a developer, I want a language‑agnostic SDK that auto‑populates the manifest during runtime, so that I don’t have to manually track AI calls.** | - SDK supports Python, Node.js, Java, and Go.<br>- SDK intercepts calls to known AI SDKs (OpenAI, Anthropic, Cohere, etc.) and logs model, version, and request payload hash.<br>- Logs are written to a local buffer and flushed to the manifest at process exit.<br>- Provides a no‑op fallback for unsupported runtimes.<br>- Unit tests demonstrate correct manifest entries for at least three AI providers. | L |
| 3 | **As a compliance officer, I want to enforce mandatory AI‑usage tags (e.g., “GDPR‑compliant”, “CCPA‑aware”) in the manifest, so that regulatory checks are automated.** | - Tag taxonomy is configurable via a central policy file.<br>- CI gate rejects builds missing required tags.<br>- UI dashboard shows compliance status per build (green/yellow/red).<br>- Audit log records who added/removed tags and when.<br>- Exportable CSV report for auditors. | M |
| 4 | **As a security auditor, I want the manifest to be cryptographically signed, so that downstream consumers can verify it hasn’t been tampered with.** | - Manifest is signed with an RSA/ECDSA key stored in the CI secret manager.<br>- Verification command (`ai‑usage‑verify verify <manifest>`) returns a clear pass/fail.<br>- Public key can be published to a well‑known URL for external verification.<br>- Signature includes a timestamp and build hash.<br>- Failure to verify blocks deployment to production. | S |

### Epic 2 – **Verification & Trust Signals for End‑Users**  
| # | User Story | Acceptance Criteria | Complexity |
|---|------------|----------------------|------------|
| 5 | **As a marketer, I want to generate a consumer‑facing AI‑usage badge for my website, so that visitors instantly see the AI components used and their trust level.** | - Badge is a embeddable HTML snippet (`<script src="…/ai‑badge.js"></script>`).<br>- Badge displays model name, provider logo, and compliance tags.<br>- Badge color reflects verification status (green = verified, red = failed).<br>- Clicking the badge opens a modal with the full manifest.<br>- Badge updates automatically when a new build is deployed. | M |
| 6 | **As an end‑user, I want to scan a QR code on a product’s packaging and view its AI‑usage report, so that I can verify the claim of AI integration.** | - QR code encodes a short URL pointing to a hosted manifest viewer.<br>- Viewer renders a human‑readable summary with model details, privacy tags, and verification signature.<br>- Viewer is responsive and accessible (WCAG 2.1 AA).<br>- Cache‑busting query param ensures the latest report is shown.<br>- Optional “Report a discrepancy” button logs feedback. | L |
| 7 | **As a journalist, I want an API endpoint that returns a machine‑readable AI‑usage certificate, so that I can programmatically validate claims in articles.** | - Endpoint: `GET /certificates/{build_id}` returns JSON with manifest, signature, and verification status.<br>- Rate‑limited (100 req/min) and requires an API key.<br>- Includes a `Cache-Control: max-age=3600` header.<br>- Returns HTTP 400 for unknown build IDs and HTTP 401 for invalid keys.<br>- Documentation with example curl commands. | S |
| 8 | **As a consumer‑rights NGO, I want to receive alerts when a previously verified product’s AI usage changes without a new verification, so that I can investigate potential misuse.** | - Change‑detection service polls the manifest repository daily.<br>- If a manifest’s hash differs from the last verified version, an email webhook is sent to subscribed NGOs.<br>- Alert includes diff summary (added/removed models, changed tags).<br>- Dashboard shows alert history per product.<br>- Opt‑in/out via a simple web form. | M |

### Epic 3 – **Integration & Ecosystem Enablement**  
| # | User Story | Acceptance Criteria | Complexity |
|---|------------|----------------------|------------|
| 9 | **As a CI/CD platform vendor, I want a plug‑in that automatically runs `ai‑usage‑verify` after each build, so that my customers get out‑of‑the‑box verification.** | - Plug‑in available for GitHub Actions, GitLab CI, and Azure Pipelines.<br>- One‑line config (`uses: axentx/ai‑usage‑verify@v1`).<br>- Plug‑in fails the pipeline on verification failure.<br>- Provides a summary artifact (`ai‑usage‑report.html`).<br>- Installation docs with screenshots for each platform. | L |
| 10 | **As a SaaS marketplace operator, I want to display each listed app’s AI‑usage badge on its store page, so that buyers can compare trust levels.** | - Marketplace can fetch badge HTML via a public endpoint (`/badge/{app_id}`).<br>- Badge includes marketplace‑specific branding overlay.<br>- Marketplace UI shows a “Verified AI” filter in the sidebar.<br>- Badge updates automatically when the app publishes a new build.<br>- Analytics track click‑through rate on the badge. | M |
| 11 | **As an open‑source contributor, I want a local Docker image that runs the verifier in an isolated environment, so that I can test my project without installing dependencies.** | - Dockerfile builds an image `axentx/ai‑usage‑verify:latest` (~150 MB).<br>- Image includes the SDK, CLI, and a sample manifest.<br>- `docker run --rm -v $(pwd):/src axentx/ai‑usage‑verify verify /src/manifest.yaml` works out of the box.<br>- README shows usage for Windows/macOS/Linux.<br>- Image is published to Docker Hub with automated CI builds. | S |
| 12 | **As a data‑privacy officer, I want a policy‑as‑code library that maps compliance tags to regional regulations, so that I can auto‑generate legal notices.** | - Library (`policy‑map.json`) maps tags → GDPR, CCPA, HIPAA, etc.<br>- CLI command `ai‑usage‑verify notice --tag privacy‑gdpr` outputs a templated notice in Markdown and HTML.<br>- Supports custom extensions via a user‑provided JSON file.<br>- Unit tests cover mapping for at least five jurisdictions.<br>- Documentation includes a “Compliance checklist” worksheet. | M |

---  

*All stories are written in Connextra format, grouped by logical epics, include 3‑5 acceptance criteria, and are sized for sprint planning (S = small, M = medium, L = large).*
